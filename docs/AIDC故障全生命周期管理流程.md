# AIDC 故障全生命周期管理流程（SOP V1.0）

> 覆盖故障发现、分级研判、应急响应、隔离定位、修复执行、恢复验证、复盘归档全流程。每一步操作均要求明确确认，确保 AIDC 高密度、高可靠性环境下的操作安全。
>
> 撰写人：孟希東

---

## 故障分级定义

分级决定响应时效和升级路径，所有后续流程依据此分级执行。

| 等级 | 定义 | 典型场景 | 响应时效 | 升级对象 |
|------|------|----------|----------|----------|
| **P0 灾难** | 整集群不可用 / 数据丢失风险 | 机柜供电中断、液冷主管路泄漏、多节点级联故障 | 5 分钟内响应 | CTO / VP Infra |
| **P1 严重** | 训练任务中断 / 部分集群不可用 | 多卡 GPU 故障、NVLink 链路中断、CDU 故障、电源架失效 | 15 分钟内响应 | 运维总监 |
| **P2 一般** | 性能降级但业务可用 | 单卡 ECC 错误、GPU 降频、单 PSU 故障、液冷水温偏高 | 30 分钟内响应 | 运维经理 |
| **P3 轻微** | 不影响当前业务 | 告警阈值预警、非关键指标异常、备件消耗预警 | 4 小时内响应 | 值班工程师 |

## Phase 1 · 故障发现与告警确认

**目标**：确认故障真实存在，排除误报，获取初始故障快照。所有告警必须在规定时间内被人工确认。

### 1.1 接收告警信号

AIOps 监控系统、Prometheus/Grafana、DCIM 平台、液冷控制系统等任一渠道触发告警。

- 确认告警来源渠道（AIOps / Prometheus / DCIM / 液冷控制器 / 人工巡检发现）
- 记录告警触发时间（精确到秒）、告警 ID、告警内容摘要
- 确认告警对象（GPU ID / 机架编号 / CDU 编号 / PSU 编号 / 网络端口）

🔴 **必须确认**：值班工程师在接收到告警后 **2 分钟内** 点击"告警确认"按钮或在运维群中回复"已接收"。超时未确认自动升级至备班工程师。

👤 责任人：当班值班工程师

### 1.2 告警真实性验证（排除误报）

通过多源交叉验证确认告警是否为真实故障，避免误报触发不必要的应急流程。

- 登录监控平台，查看告警指标的历史趋势，判断是瞬时毛刺还是持续异常
- 交叉验证：至少使用两个独立数据源确认异常（如 nvidia-smi + Prometheus + DCIM）
- 对于 GPU 告警：执行 `nvidia-smi -q -i [GPU_ID]` 获取当前状态快照
- 对于液冷告警：检查 CDU 控制面板的流量、压力、温度实时读数
- 对于供电告警：查看 PDU/PSU 监控面板的电压、电流、功率读数
- 对于网络告警：检查交换机端口状态和错误计数器

🔴 **必须确认**：在故障工单中明确标注"已验证为真实故障"或"确认为误报"。误报需记录误报原因并调整告警阈值。

ℹ️ **判定为误报时**：关闭告警 → 记录误报原因 → 提交告警规则优化建议 → 流程终止。

### 1.3 采集故障现场快照

在任何操作介入之前，完整保留故障现场状态，作为后续根因分析和复盘的关键证据。

- GPU 快照：`nvidia-smi -q` 完整输出、`dmesg | grep -i gpu`、XID 错误日志
- 液冷快照：CDU 当前参数（流量/压力/进出口温度/水质指标）截图
- 供电快照：PSU 状态、母线温度读数、UPS 电池状态
- 网络快照：NVLink 错误计数（`nvidia-smi nvlink -e`）、IB 端口状态
- 系统日志：dmesg、/var/log/syslog、应用层训练任务日志
- 环境快照：机房温湿度、漏液传感器状态

🔴 **必须确认**：所有快照数据已保存至故障工单附件，并注明采集时间。**严禁在快照采集完成前执行任何修复操作。**

👤 责任人：当班值班工程师

### 1.4 创建故障工单

在 ITSM 系统中创建正式故障工单，后续所有操作均挂靠此工单记录。

- 工单必填字段：故障标题、发现时间、告警来源、故障对象、初步现象描述
- 关联告警 ID，附加步骤 1.3 中的所有快照附件
- 设置初始优先级（可在步骤 2 中调整）

✅ **确认**：故障工单已创建，工单号已同步至运维沟通群。

## Phase 2 · 故障分级与影响研判

**目标**：准确评估故障严重程度和影响范围，确定响应级别和升级路径。

### 2.1 故障域判定

确定故障所属的运维域，决定后续由哪个专业团队主导处理。

- 判定故障域：GPU 算力 / 液冷系统 / 供电系统 / 网络 / 存储 / 基础设施
- 判断是否涉及跨域故障（如液冷异常导致 GPU 降频，属于液冷+GPU 联合故障）
- 记录受影响的具体设备清单（GPU 编号、机架编号、节点 IP 等）

🔴 **必须确认**：故障域已明确标注在工单中。跨域故障需同时通知所有相关域的负责人。

### 2.2 影响范围评估

评估故障对业务的实际影响和潜在扩散风险。

- 确认受影响的训练/推理任务数量和任务优先级
- 确认受影响的 GPU 数量占集群总量的百分比
- 评估故障是否有扩散风险（如液冷泄漏可能蔓延、PSU 故障可能过载其他 PSU）
- 确认是否有冗余保护仍在生效（如 N+1 电源冗余、NVLink 路径冗余）
- 确认数据安全状态：最近一次 checkpoint 时间，是否存在数据丢失风险

🔴 **必须确认**：影响范围已量化记录：受影响任务数 / GPU 数 / 节点数 / 预计业务损失时间。

### 2.3 故障定级

根据影响范围和严重程度，按照分级定义表确定故障等级。

- 对照本文档开头的《故障分级定义》表，确定故障等级（P0/P1/P2/P3）
- P0/P1 级别需立即电话通知对应升级对象
- P0 级别需同时启动应急作战室（War Room）

🔴 **必须确认**：故障等级已在工单中更新。P0/P1 级别已电话确认升级对象已知晓（非仅消息通知）。

👤 责任人：值班工程师定级 → 运维经理复核确认

### 2.4 通知相关方

按故障等级通知对应的干系人，确保信息同步。

- 通知受影响业务方的技术负责人，告知预计影响时间
- P0/P1：通知运维总监、基础设施负责人、业务方 VP
- 涉及液冷/供电的物理故障：通知现场驻场工程师和设备供应商
- 涉及 GPU 硬件故障：通知 NVIDIA TAM 或技术支持

✅ **确认**：所有应通知人员已通知到位，在工单中记录通知时间和确认回复。

## Phase 3 · 应急响应与任务保护

**目标**：在故障修复之前，最大程度保护正在运行的任务和数据，防止故障扩散。

### 3.1 评估是否需要紧急保存 Checkpoint

判断当前故障是否可能导致训练任务数据丢失，决定是否需要紧急触发 checkpoint。

- 确认受影响节点上正在运行的训练任务列表和当前 epoch/step
- 查询最近一次成功 checkpoint 的时间和 step 号
- 如果距上次 checkpoint 超过 1 小时或超过 1000 steps，评估紧急保存的可行性
- 如果故障节点仍可通信，尝试触发紧急 checkpoint 保存

🔴 **必须确认**：已评估 checkpoint 状态。选择项：① 已触发紧急 checkpoint 并确认保存成功 ② checkpoint 保存失败，记录原因 ③ 无需紧急保存（最近 checkpoint 足够新）④ 节点不可达，无法保存。

### 3.2 训练任务迁移或暂停

将受影响节点上的任务迁移到健康节点，或有序暂停任务。

- 确认集群中是否有足够的空闲健康 GPU 可供任务迁移
- 如可迁移：通过调度系统将任务重新调度到健康节点，从最新 checkpoint 恢复
- 如不可迁移：有序暂停任务（发送 graceful shutdown 信号，等待当前 step 完成）
- **严禁直接 kill -9 训练进程**，除非节点已完全失联

🔴 **必须确认**：所有受影响任务的当前状态已记录：① 已迁移（目标节点 + 恢复 step）② 已暂停（暂停时间）③ 已丢失（丢失的 step 范围）。通知业务方确认任务状态。

### 3.3 故障扩散防护

采取措施防止故障从受影响设备扩散到健康设备。

- GPU 故障：将故障 GPU 标记为不可调度（drain node），避免新任务分配到该节点
- 液冷泄漏：立即关闭受影响管路段的阀门，启动漏液应急预案
- 供电异常：如单 PSU 故障且 N+1 冗余仍在，持续监控其余 PSU 负载，设置加强告警
- 供电异常：如母线温度异常升高，评估是否需要降低机架负载
- 网络故障：隔离故障端口/链路，确认流量已切换至冗余路径

🔴 **必须确认**：故障隔离措施已执行，健康设备未受波及。在工单中记录已采取的防护措施。

⚠️ **液冷泄漏特别提醒**：如检测到液冷泄漏，必须立即断开受影响机架的供电（先断电再处理泄漏），并疏散机房内人员至安全区域。此操作优先级高于一切。

👤 责任人：值班工程师执行 → 运维经理确认

## Phase 4 · 故障隔离与根因定位

**目标**：精确定位故障根本原因，为制定修复方案提供依据。

### 4.1 分域深度诊断

根据故障域执行对应的深度诊断流程。

**GPU 故障诊断：**
- 检查 XID 错误码（`dmesg | grep -i xid`），对照 NVIDIA XID 错误码表定位故障类型
- 检查 ECC 错误计数（`nvidia-smi --query-gpu=ecc.errors.* --format=csv`）
- 检查 Retired Pages 和 Remapped Rows 数量
- 检查 GPU 降频原因（`nvidia-smi -q -d PERFORMANCE`）
- 运行 DCGM 诊断（`dcgmi diag -r 3`）进行 Level 3 全面健康检查

**液冷系统诊断：**
- 检查 CDU 运行日志，确认故障发生时的流量/压力/温度曲线
- 检查冷却液水质报告（pH值、电导率、颗粒物计数）
- 检查快接头连接状态和管路外观（是否有渗漏痕迹）
- 检查泵组运行状态和振动数据

**供电系统诊断：**
- 检查电源架各 PSU 模块的输入输出电压/电流/温度
- 检查铜母线各段的温度读数和红外热成像（如有）
- 检查 UPS 电池组状态和最近一次自检结果

**网络诊断：**
- 检查 NVLink 各链路的错误计数和吞吐量
- 检查 InfiniBand 端口的 SymbolError、LinkRecover 计数器
- 检查光模块的收发光功率是否在正常范围

🔴 **必须确认**：诊断数据已完整采集并附加到工单。每项诊断结果标注"正常"或"异常+具体数值"。

### 4.2 关联分析与根因推断

综合多维度诊断数据，分析因果关系，推断根本原因。

- 按时间线排列所有异常事件，确认故障发生的先后顺序
- 检查是否存在关联故障（如液冷温度升高 → GPU 降频 → 训练任务超时）
- 对照历史相似故障案例库，查找是否有已知解决方案
- 如无法自行定位，联系设备厂商技术支持（NVIDIA / 液冷供应商 / 电源供应商）

🔴 **必须确认**：根因已明确或缩小到有限范围。在工单中记录：根因描述 + 判断依据 + 置信度（确定/高度疑似/待确认）。

### 4.3 根因确认评审

对于 P0/P1 级别故障，根因判断需要经过评审确认。

- P0/P1：召集相关域专家进行根因评审会议（可线上）
- 评审内容：诊断数据是否充分、根因推断是否合理、是否遗漏其他可能性
- P2/P3：由运维经理审核确认即可

🔴 **必须确认**：根因已通过评审/审核确认。工单中记录确认人和确认时间。

👤 责任人：故障域专家定位 → 运维经理/总监审核确认

## Phase 5 · 修复方案制定与执行

**目标**：制定安全可控的修复方案并严格执行，确保修复过程不引入新的问题。

### 5.1 制定修复方案

根据根因制定具体修复方案，包括操作步骤、风险评估和回滚计划。

- 明确修复方式：软件修复（重启服务/重载驱动/更新固件）或硬件更换
- 列出详细操作步骤（精确到每一条命令或每一个物理操作动作）
- 评估修复操作的风险：是否需要停机、是否影响同机架其他设备
- 制定回滚方案：如果修复操作失败，如何回退到修复前状态
- 确认所需的备件/工具是否已准备就绪（如需硬件更换）
- 确认修复操作的预计耗时

🔴 **必须确认**：修复方案文档已撰写并包含：操作步骤 + 风险评估 + 回滚方案 + 所需备件 + 预计耗时。

### 5.2 修复方案审批

修复方案执行前必须经过对应级别的审批。

- P0/P1：运维总监审批（紧急情况可电话口头审批，事后补签）
- P2：运维经理审批
- P3：高级工程师审批
- 涉及硬件更换：需同时获得资产管理部门确认备件出库
- 涉及液冷/供电操作：需同时获得设施管理团队确认

🔴 **必须确认**：修复方案已获得对应级别审批。工单中记录审批人、审批时间、审批意见。**未经审批严禁执行修复操作。**

### 5.3 修复前预检

执行修复操作前的最终确认清单，确保环境安全。

- 再次确认故障设备已正确隔离，不承载业务流量
- 确认修复操作不会影响同机架/同行的其他健康设备
- 硬件操作前：确认已穿戴防静电手环，工具齐全
- 液冷相关操作：确认已关闭对应管路阀门，准备好接液容器和吸液材料
- 供电相关操作：确认已断开相关回路，使用万用表验证无带电
- 确认有第二人在场（涉及高压/液冷操作必须双人作业）

🔴 **必须确认**：预检清单全部通过。操作人员和复核人员双签确认。

⚠️ **安全红线**：高压供电操作和液冷管路操作必须双人作业，严禁单人操作。操作期间保持通讯畅通。

### 5.4 执行修复操作

严格按照审批通过的方案逐步执行，每一步操作后确认结果。

- 按方案中的步骤顺序逐条执行
- 每执行一步，确认该步结果正常后再进行下一步
- 实时在工单中记录每一步的执行时间和执行结果
- 如任一步骤结果异常，立即暂停，评估是否触发回滚
- 软件操作示例：重载 GPU 驱动 → 确认驱动加载成功 → 确认 GPU 可被识别 → 下一步
- 硬件操作示例：拔出故障托盘 → 确认机架状态正常 → 插入新托盘 → 确认物理连接 → 上电 → 确认启动

🔴 **必须确认**：每一步操作的执行结果已记录。修复操作整体完成后，操作人员在工单中标注"修复操作已完成"。

### 5.5 修复后即时检查

修复操作完成后，立即检查修复效果和是否引入新问题。

- GPU 修复后：执行 `nvidia-smi` 确认 GPU 可见 → ECC 错误清零 → 运行 `dcgmi diag -r 2`
- 液冷修复后：检查管路压力恢复 → 流量正常 → 无泄漏 → 温度开始回落
- 供电修复后：确认 PSU 输出电压/电流正常 → 母线温度正常 → 冗余状态恢复
- 网络修复后：确认端口 UP → 错误计数器归零 → 带宽测试通过
- 检查同机架其他设备是否受到影响

🔴 **必须确认**：即时检查全部通过。如有任何异常项，记录异常并评估是否需要进一步处理或回滚。

👤 责任人：操作工程师执行 → 复核工程师确认

## Phase 6 · 恢复验证与业务回切

**目标**：通过多级验证确认故障已彻底修复，安全地恢复业务。

### 6.1 功能性验证

验证修复后的设备功能是否完全正常。

- GPU：运行 DCGM Level 3 完整诊断（`dcgmi diag -r 3`），包括显存压力测试
- GPU：运行 gpu-burn 或 NCCL allreduce 测试，验证计算和通信能力
- NVLink：运行 `nvidia-smi nvlink -gt d` 确认带宽恢复到标称值
- 液冷：持续观察 30 分钟，确认温度、流量、压力稳定在正常范围
- 供电：确认负载恢复后各指标稳定，持续观察 15 分钟

🔴 **必须确认**：功能性验证全部通过，测试报告已附加到工单。任何一项不通过则返回 Phase 5 重新排查。

### 6.2 压力测试验证

在接近生产负载的条件下验证设备的稳定性。

- 运行与生产环境相似的测试任务，持续至少 1 小时
- 监控期间的 GPU 温度、功耗、ECC 错误、NVLink 错误是否出现异常
- 监控液冷系统在满载下的散热能力是否正常
- 确认测试期间无新告警产生

🔴 **必须确认**：压力测试持续 ≥1 小时且全程无异常。测试期间的监控数据截图已附加到工单。

### 6.3 解除隔离，恢复可调度状态

验证通过后，将设备从维护状态恢复为可调度状态。

- 在调度系统中将节点从 drain/maintenance 状态恢复为 schedulable
- 确认节点状态在调度系统中显示为 Ready
- 确认监控系统已恢复对该节点的正常监控（非静默状态）

🔴 **必须确认**：节点已恢复可调度状态。调度系统中状态显示正常。

### 6.4 业务回切

将之前迁移或暂停的业务任务切回修复后的设备。

- 与业务方沟通确认回切时间窗口
- 优先调度低优先级任务到修复后的节点，观察运行状态
- 低优先级任务运行稳定后，逐步回切高优先级训练任务
- 确认所有回切的任务运行正常，训练 loss 曲线无异常

🔴 **必须确认**：业务方已确认任务恢复正常。工单中记录回切完成时间和业务方确认人。

### 6.5 持续观察期

业务回切后进入持续观察期，确认故障不再复发。

- P0/P1 故障：观察期 72 小时
- P2 故障：观察期 48 小时
- P3 故障：观察期 24 小时
- 观察期内设置加强监控（降低告警阈值或缩短采集间隔）
- 每日检查一次关键指标（ECC 错误、温度、功耗、NVLink 错误）是否异常

🔴 **必须确认**：观察期结束，期间无异常复发。在工单中标注"观察期通过，故障确认修复"。如观察期内复发，重新进入 Phase 4。

👤 责任人：值班工程师观察 → 运维经理确认关闭

## Phase 7 · 复盘归档与持续改进

**目标**：总结经验教训，更新知识库和运维流程，防止同类故障再次发生。

### 7.1 故障复盘会议

P0/P1 故障必须召开正式复盘会议，P2/P3 书面复盘。

- P0/P1：故障修复后 3 个工作日内召开复盘会议
- 参会人员：故障处理全流程参与人员 + 运维管理层 + 业务方代表
- 复盘内容：故障时间线回顾、根因分析、处理过程评估、改进建议
- 重点评估：告警是否及时、响应是否达标、操作是否规范、有无可优化环节

✅ **确认**：复盘会议已召开（P0/P1）或书面复盘已完成（P2/P3），会议纪要已归档。

### 7.2 故障报告撰写

撰写正式的故障报告（Post-Mortem / RCA Report）。

- 报告必须包含：故障概述、影响范围和时长、详细时间线、根因分析、修复措施、改进 Action Items
- 时间线精确到分钟，记录每个关键决策节点和操作动作
- 明确列出改进项、责任人和完成期限
- P0 故障报告需发送至技术管理层

✅ **确认**：故障报告已完成并经运维经理审核。报告已归档至知识库。

### 7.3 知识库与监控优化

将故障经验转化为可复用的知识和系统改进。

- 更新故障案例库：录入故障现象、根因、解决方案，建立索引标签
- 更新运维手册 / SOP：如果现有流程存在不足，补充或修订操作步骤
- 优化告警规则：调整误报/漏报的告警阈值和关联规则
- 更新 AIOps 模型：将本次故障数据用于训练预测模型
- 评估是否需要增加新的监控指标或巡检项

✅ **确认**：知识库已更新，告警规则已优化（如需要），运维手册已修订（如需要）。

### 7.4 改进项跟踪与闭环

确保复盘中提出的改进措施被落实。

- 将每项改进措施创建为独立的跟踪工单，指定责任人和完成期限
- 运维经理每周跟踪改进项进度
- 改进项完成后需验证效果（如新增的告警是否能在模拟测试中正确触发）

✅ **确认**：所有改进项已创建跟踪工单。定期回顾直至全部闭环。

### 7.5 故障工单关闭

确认所有流程环节完成后，正式关闭故障工单。

- 检查工单信息完整性：所有阶段的确认记录、附件、审批记录是否齐全
- 确认观察期已通过，故障未复发
- 确认故障报告已归档
- 确认改进项已创建跟踪
- 计算故障关键指标：MTTD（发现时间）、MTTR（恢复时间）、业务影响时长

🔴 **最终确认**：运维经理审核工单完整性后关闭。工单关闭后记录 MTTD / MTTR / 业务影响时长，纳入月度运维报告统计。

👤 责任人：运维经理关闭工单
