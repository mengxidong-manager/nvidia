# Dell 服务器 RAID 故障恢复操作手册

> 覆盖 RAID 0 / 1 / 5 / 10 四种级别的故障识别、应急处理、硬盘更换、阵列重建完整流程。以 Dell PowerEdge 服务器（PERC 控制器）为例。
>
> 撰写人：孟希東

---

## RAID 级别速览与容灾能力

| RAID | 原理 | 最少磁盘 | 可用容量 | 允许坏盘 | 坏盘后果 |
|------|------|----------|----------|----------|----------|
| RAID 0 | 条带化 | 2 | 100% | **0** | ❌ 全部数据丢失，不可恢复 |
| RAID 1 | 镜像 | 2 | 50% | **1** | ✅ 可降级运行，更换后自动重建 |
| RAID 5 | 条带+校验 | 3 | (N-1)/N | **1** | ✅ 可降级运行，更换后自动重建 |
| RAID 10 | 镜像+条带 | 4 | 50% | **每组1块** | ✅ 可降级运行，更换后自动重建 |

> ⚠️ **核心原则**：RAID 不等于备份。RAID 提供的是冗余和可用性，不能替代定期数据备份。在进行任何 RAID 恢复操作前，务必先确认最近的备份状态。

## 1. 故障发现与识别

适用于所有 RAID 级别，是恢复流程的第一步。

### 物理指示灯检查

服务器前面板或硬盘托架指示灯异常：琥珀色闪烁 = 故障预警（Predictive Failure），琥珀色常亮 = 硬盘故障（Failed），绿色闪烁 = 正常读写，绿色常亮 = 在线正常。

### iDRAC 告警检查

登录 Dell iDRAC 管理界面（默认 `https://<iDRAC-IP>`），进入 Storage → Physical Disks，查看硬盘状态。关注状态为 "Failed" 或 "Degraded" 的磁盘。

```
Dashboard → Storage → Physical Disks → 查看 Status 列
```

### 操作系统层面检查

```bash
# 使用 Dell OMSA 或 perccli 命令行工具检查
perccli /c0 show                # 查看控制器和虚拟磁盘状态
perccli /c0/eall/sall show      # 查看所有物理磁盘状态
perccli /c0/v0 show             # 查看虚拟磁盘详情
```

### 确认故障信息并记录

记录以下关键信息：故障磁盘的槽位号（Slot）、序列号（S/N）、型号、RAID 级别、虚拟磁盘当前状态（Degraded / Failed / Offline）。截图或导出 iDRAC 日志备查。

> ℹ️ **PERC 控制器状态说明**：Virtual Disk 状态 "Optimal" = 正常，"Degraded" = 降级（有冗余盘故障但仍可运行），"Failed" = 阵列失败，"Rebuilding" = 正在重建。

## 2. RAID 0 故障恢复

> 🔴 **RAID 0 没有容灾能力。** 任何一块硬盘故障，整个阵列的数据全部丢失且无法通过 RAID 机制恢复。恢复只能依赖备份。

### 恢复流程

1. **确认数据丢失**：RAID 0 阵列中任一磁盘故障后，虚拟磁盘状态变为 "Failed"。此时阵列中的数据已不可读取，确认数据丢失。

2. **评估数据恢复需求**：如果数据极其重要且没有备份，停止一切操作，不要拔盘、不要重建，联系专业数据恢复公司。他们可能通过底层磁盘分析恢复部分数据。

3. **更换故障硬盘**：关机 → 更换故障硬盘 → 使用相同规格的新盘（建议同型号同容量）。

4. **重新创建 RAID 0 阵列**：开机 → 进入 PERC BIOS（Ctrl+R 或 F2 → Device Settings → RAID Controller）→ 删除旧的虚拟磁盘 → 使用所有磁盘创建新的 RAID 0 虚拟磁盘。

```bash
# 或使用 perccli 命令行
perccli /c0/v0 delete                        # 删除旧虚拟磁盘
perccli /c0 add vd r0 drives=32:0,32:1       # 创建新 RAID 0
```

5. **重装系统并从备份恢复数据**：安装操作系统 → 从最近的备份恢复数据。如果 RAID 0 用于非关键数据（如临时缓存），可直接投入使用。

> ⚠️ **建议**：生产环境不要使用 RAID 0。如果必须使用，务必配合高频备份策略。

## 3. RAID 1 故障恢复

RAID 1 · 镜像 · 允许坏 1 块盘 · 可在线重建

### 场景一：单盘故障（最常见）

1. **确认阵列状态为 Degraded**：iDRAC 或 perccli 显示虚拟磁盘状态为 "Degraded"，一块物理盘状态为 "Failed"。此时系统仍在运行，数据安全。

```bash
perccli /c0/v0 show    # 确认 State: Degraded
```

2. **记录故障盘信息**：记录故障磁盘的 Slot 号、Enclosure ID、序列号。确认新盘型号和容量 ≥ 故障盘。

```bash
perccli /c0/eall/sall show    # 查看所有物理盘，找到 State: Failed 的盘
```

3. **热插拔更换硬盘（无需关机）**：Dell PowerEdge 支持热插拔。直接拔出故障硬盘（琥珀灯亮的那块），插入新盘。等待约 10 秒，新盘指示灯开始绿色闪烁。

4. **确认自动重建（Auto Rebuild）**：Dell PERC 控制器默认开启自动重建。新盘插入后，控制器会自动将好盘的数据镜像到新盘。

```bash
perccli /c0/v0 show                  # 确认 State: Rebuilding
perccli /c0/eall/sall show rebuild   # 查看重建进度
```

5. **如未自动重建，手动触发**：

```bash
perccli /c0/e32/s1 set good      # 先将新盘设为 Unconfigured Good
perccli /c0/e32/s1 start rebuild # 手动开始重建
```

6. **等待重建完成**：重建时间取决于磁盘容量和系统负载。1 TB 硬盘通常需要 2-4 小时。重建期间系统正常运行，但 I/O 性能会有所下降。

7. **确认恢复完成**：虚拟磁盘状态从 "Rebuilding" 变为 "Optimal"。物理盘状态从 "Rebuilding" 变为 "Online"。

```bash
perccli /c0/v0 show    # 确认 State: Optimal ✓
```

### 场景二：两块盘同时故障（阵列失败）

阵列状态变为 Failed：RAID 1 只有两块盘，两块都坏则阵列失败，数据丢失。处理方式同 RAID 0：更换两块盘 → 重建阵列 → 从备份恢复。

## 4. RAID 5 故障恢复

RAID 5 · 条带+分布式校验 · 允许坏 1 块盘 · 可在线重建

### 场景一：单盘故障（Degraded）

1. **确认阵列降级**：虚拟磁盘状态 "Degraded"，一块物理盘 "Failed"。系统正常运行，数据通过校验数据可正常读取。

```bash
perccli /c0/v0 show    # State: Degraded
```

2. **立即降低系统 I/O 负载（建议）**：降级状态下如果再坏一块盘，数据将全部丢失。建议暂停非关键业务、减少写入操作，尽快完成磁盘更换。

3. **记录故障盘信息并准备新盘**：确认故障盘 Slot/Enclosure，准备同规格新盘（容量 ≥ 故障盘）。

```bash
perccli /c0/eall/sall show    # 找到 Failed 的盘
```

4. **热插拔更换硬盘**：拔出故障盘 → 插入新盘 → 等待 10 秒确认新盘被识别。

5. **确认自动重建开始**：PERC 控制器自动使用校验数据 + 其余盘数据计算并写入新盘。

```bash
perccli /c0/v0 show                  # State: Rebuilding
perccli /c0/eall/sall show rebuild   # 查看进度百分比
```

6. **如未自动重建，手动触发**：

```bash
perccli /c0/e32/s2 set good
perccli /c0/e32/s2 start rebuild
```

7. **监控重建进度**：RAID 5 重建比 RAID 1 慢（需要计算校验），大容量磁盘可能需要 4-12 小时甚至更长。期间避免重启服务器。

```bash
# 每隔 30 分钟检查一次进度
perccli /c0/eall/sall show rebuild
```

8. **确认重建完成**：虚拟磁盘 → "Optimal"，新物理盘 → "Online"。

```bash
perccli /c0/v0 show    # State: Optimal ✓
```

### 🔴 场景二：重建期间又坏一块盘（双盘故障）

RAID 5 仅允许坏 1 块盘。重建期间所有磁盘负载极高，是最脆弱的时期。如果此时再坏一块盘，阵列失败，全部数据丢失。因此 RAID 5 重建前务必确认备份可用，重建期间不要做大量 I/O 操作。大容量磁盘（>4TB）的 RAID 5 重建风险较高，生产环境建议使用 RAID 6 或 RAID 10。

## 5. RAID 10 故障恢复

RAID 10 · 先镜像后条带 · 每个镜像组允许坏 1 块 · 可在线重建

### 场景一：单盘故障（最常见）

1. **确认阵列降级**：虚拟磁盘 "Degraded"，某个镜像组中的一块盘 "Failed"。该镜像组的另一块盘仍在正常工作，数据安全。

```bash
perccli /c0/v0 show          # State: Degraded
perccli /c0/v0 show all      # 查看哪个 Span（镜像组）受影响
```

2. **确认故障盘所属镜像组**：RAID 10 由多个镜像组（Span）组成。确认故障盘属于哪个 Span，以及该 Span 的另一块盘是否正常。

3. **热插拔更换硬盘**：拔出故障盘 → 插入新盘（同规格）→ 等待控制器识别。

4. **自动重建**：控制器将同镜像组的好盘数据镜像到新盘。RAID 10 的重建速度比 RAID 5 快很多，因为只需复制一块盘的数据（不需要计算校验）。

```bash
perccli /c0/eall/sall show rebuild    # 查看重建进度
```

5. **确认重建完成**：虚拟磁盘 → "Optimal"。

```bash
perccli /c0/v0 show    # State: Optimal ✓
```

### 场景二：不同镜像组各坏一块盘

如果 4 块盘（2 个镜像组），Span 0 坏了一块、Span 1 也坏了一块——阵列仍可运行（Degraded），因为每个镜像组还有一块好盘。按上述流程依次更换两块故障盘，控制器会逐个重建。

### 🔴 场景三：同一镜像组两块盘都坏

如果同一个 Span 的两块盘都故障，该 Span 的数据丢失 → 整个 RAID 10 阵列失败。恢复方式：更换所有故障盘 → 重建阵列 → 从备份恢复数据。

## 6. 通用硬盘更换流程（Dell 热插拔）

适用于所有支持热插拔的 Dell PowerEdge 服务器。

1. **准备工作**：确认新盘型号、容量、接口与故障盘一致（建议同品牌同型号）。新盘容量必须 ≥ 故障盘。准备防静电手环。

2. **定位故障硬盘**：通过 iDRAC 或 perccli 确认故障盘的物理槽位号。在服务器前面板找到对应槽位（琥珀色指示灯）。可以通过 iDRAC 触发 LED 闪烁辅助定位：

```bash
perccli /c0/e32/s2 start locate    # 让目标盘指示灯闪烁
```

3. **拔出故障硬盘**：按下硬盘托架上的释放按钮 → 拉出硬盘托架 → 轻轻拔出。注意不要晃动相邻硬盘。

4. **安装新硬盘**：将新硬盘安装到托架上（如需从旧托架拆换）→ 对准导轨推入槽位 → 听到"咔嗒"声确认卡扣到位。

5. **等待控制器识别**：等待 10-30 秒。新盘指示灯变为绿色闪烁表示已被识别。如果指示灯不亮，重新插拔确认物理连接。

6. **确认重建自动启动**：对于 RAID 1/5/10，控制器默认自动开始重建。通过 iDRAC 或 perccli 确认重建进度。

```bash
perccli /c0/eall/sall show rebuild
```

7. **验证重建完成**：等待重建 100% 完成 → 虚拟磁盘状态恢复 "Optimal" → 记录操作日志。

### 重建时间参考

| 磁盘容量 | RAID 1 | RAID 5 | RAID 10 |
|----------|--------|--------|---------|
| 500 GB | 1-2 小时 | 2-3 小时 | 1-2 小时 |
| 1 TB | 2-4 小时 | 4-6 小时 | 2-4 小时 |
| 4 TB | 6-10 小时 | 10-20 小时 | 6-10 小时 |
| 8 TB+ | 12-20 小时 | 20-40+ 小时 | 12-20 小时 |

> \* 时间为估算值，实际取决于磁盘类型（SSD/HDD）、控制器型号、系统 I/O 负载。SSD 重建时间通常为 HDD 的 1/5 到 1/10。

## 7. 注意事项与最佳实践

### 🟢 应该做

- 更换前确认最近备份可用
- 使用同品牌同型号同容量的替换盘
- 重建期间降低系统 I/O 负载
- 重建完成后执行一致性检查（Consistency Check）
- 配置 Hot Spare 热备盘加速自动恢复
- 定期检查 SMART 数据，预防性更换老化磁盘
- 记录每次更换操作到运维日志

### 🔴 不应该做

- 重建期间重启服务器（可能中断重建）
- 同时拔出多块正常硬盘
- 使用容量小于故障盘的替换盘
- 在不确认故障盘位置时随意拔盘
- RAID 降级期间执行大规模数据迁移
- 忽略 Predictive Failure 预警不处理
- 误将 RAID 当作备份方案

### Hot Spare 热备盘配置（强烈推荐）

热备盘是预先指定的空闲磁盘，当 RAID 中有盘故障时，控制器自动使用热备盘开始重建，无需人工更换。可将故障发现到重建开始的时间从"人工响应时间"缩短到"秒级"。

```bash
# 配置全局热备盘（对所有虚拟磁盘生效）
perccli /c0/e32/s7 add hotsparedrive

# 配置专用热备盘（仅对指定虚拟磁盘生效）
perccli /c0/e32/s7 add hotsparedrive dgs=0
```

### 一致性检查（重建完成后执行）

重建完成后，建议运行一致性检查确保所有条带和校验数据正确。

```bash
perccli /c0/v0 start cc    # 启动一致性检查
perccli /c0/v0 show cc     # 查看检查进度
```

> ✅ **恢复流程总结**：识别故障 → 确认 RAID 级别和影响 → 记录故障盘信息 → 准备新盘 → 热插拔更换 → 确认自动重建 → 监控重建进度 → 验证 Optimal → 运行一致性检查 → 记录操作日志。
